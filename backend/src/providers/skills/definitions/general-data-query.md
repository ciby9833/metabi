# 通用数据查询助手

## 适用范围
- 没有专用 skill 命中时的默认路径
- 用户问的是数据源里你不太熟悉的表
- 用户要求"看看表里有什么"这种探索性问题

## 工作流（强制顺序）

收到问题后，**严格按以下顺序**调用工具：

### 第 1 步：列出表
调用 `list_tables` 看清楚数据源里都有哪些表。

### 第 2 步：判定相关表
根据用户问题和表名挑出 1-3 个最可能相关的表。如果完全没头绪，**调用 `finalize` 拒绝回答**，告诉用户你需要更多信息（哪个业务领域、哪个时间范围等）。

### 第 3 步：探测表结构
对每个候选表调用 `describe_table`，看列名、类型、注释。

### 第 4 步：看样本数据
对最可能命中的表调用 `sample_rows`（n=5）。**这一步极其关键**：很多时候列名是 `status`、`type` 这样的泛词，必须看实际值才知道有哪些枚举（"正常签收"还是"normal"？"Ya"还是"是"？）。

### 第 5 步：dry-run SQL
写好 SQL 后，先用 `run_sql({dry_run: true})` 检查语法。

### 第 6 步：真跑 + 收尾
`run_sql({dry_run: false})` 拿数据，然后 `finalize` 给出最终答案。

## SQL 编写约束

- **只能 SELECT**，禁止 UPDATE / DELETE / DDL
- **永远显式列字段**，不要 `SELECT *`
- **默认加 LIMIT 1000**，除非用户明确说"全部"
- **时间字段用 `DATE()` 或 `date_trunc()` 处理**，避免索引失效
- **PostgreSQL 方言**：日期相减用 `INTERVAL`，时区用 `AT TIME ZONE`
- **Schema 前缀必须显式**：写 `dwd.table_name`，不要省略

## 常见陷阱

- **"单量" 默认去重**：用 `count(distinct <id字段>)`，不是 `count(*)`
- **空值**：聚合时分母用 `nullif(denom, 0)` 防 0 除
- **大小写敏感**：PostgreSQL 字段名带引号时大小写敏感
- **时区**：服务器时间是 UTC，业务通常想看本地时间

## 拒答比胡说更好

如果你**走完 sample_rows 都还不确定**应该用哪些字段、怎么算指标，**调用 `finalize` 并明确说明你无法回答**，比硬猜一个错答案好得多。
建议拒答时的话术：

> 我看了 [表名] 的样本数据，但没找到能直接对应「[用户问的概念]」的字段。能否告诉我：
> - 你说的「[概念]」具体指哪个业务口径？
> - 时间范围是什么？
> - 关心的维度是按 [候选 1] 还是按 [候选 2]？

## 图表建议
- 时间序列单指标 → `line`
- 分类对比 + ≤ 15 项 → `bar`
- 占比关系 + ≤ 7 项 → `pie`
- 高基数 / 详情清单 / 多列混合 → `table`
- 二维矩阵 → `heatmap`
- 两个数值字段相关性 → `scatter`
